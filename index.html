preact元素生成原理

发表于 2017-08-19

preact元素生成主要分为两个步骤：

  1. jsx 解析为特定格式的 Object（虚拟dom）其格式为：{nodename, attribute, children}
  2. 渲染vdom并插入到挂载点

v-dom生成

jsx解析借助了babel-plugin-transform-react-jsx，具体转化代码如下

let items = ['textA', 'textB', 'textC'];
let list = items => items.map( text => <li>{ text }</li>);
let foo = (
    <div id="foo">
        zhi zhi zhi!
        <ul>
            {list(items)}
        </ul>    
    </div>
);

//**************    解析后    ************
let items = ['textA', 'textB', 'textC'];
let list = items => items.map(text => Object(__WEBPACK_IMPORTED_MODULE_0__preact__["a" /* h */])(
    'li',
    null,
    text
));
let foo = Object(__WEBPACK_IMPORTED_MODULE_0__preact__["a" /* h */])(
    'div',
    { id: 'foo' },
    'zhi zhi zhi!',
    Object(__WEBPACK_IMPORTED_MODULE_0__preact__["a" /* h */])(
        'ul',
        null,
        list(items)
    )
);

从代码可以看出，babel会解析JSX的dom数据作为参数，嵌套调用__WEBPACK_IMPORTED_MODULE_0__preact__["a" /* h */]，这个函数默认是babel提供的dom函数，也可以通过配置pragma参数自定义这个函数（本文自定义了h函数），其作用是预处理输入的dom信息，返回特定格式的对象。

//***********      h函数      **************
// h函数代表 hyperscript （hypertext + javascript）
const h = function (nodeName, attr, ...args) {
    let children = args.length ? [].concat(...args) : null;
    return { nodeName, attr, children };
}

渲染vdom并插入到挂载点

接下来就是定义vdom解析函数了。

/**
 * @param {Object} vdom v-dom
 * @return {Object} dom片段
 **/
const createDOM = function (vnode) {
    if(typeof vnode === 'string') return document.createTextNode(vnode);

    let node = document.createElement(vnode.nodeName);

    let attributes = vnode.attributes || {};
    Object.keys(attributes).forEach(key => node.setAttribute(key, attributes[key]));

    (vnode.children || []).forEach(child => node.appendChild(createDOM(child)));

    return node;
}

/**
 * @param {Object} vdom v-dom
 * @param {Object} target 挂载点
 **/
const render = function(vdom, target) {
    const dom = createDOM(vdom);
    target.appendChild(dom);
}

//*****************    再调用看看    **********************
let items = ['textA', 'textB', 'textC'];
let list = items => items.map( text => <li>{ text }</li>);
let foo = (
    <div id="foo">
        mice..zhi zhi zhi!
        <ul>
            {list(items)}
        </ul>    
    </div>
);

render(foo, document.body);

99%的前端初学者都会搞混的offset...

发表于 2017-07-30

前端学习过程中会经常遇到offsetX、offsetLeft、clientX之类的概念，如果没有作系统性的总结，就很容易搞混。本文主要基于MDN的定义和自身的实践总结，其中有兼容性问题的属性会用()在后面注明。

话休饶舌，下面让小生来给大家梳理一下。

与offset相关的属性可以分为两类：

mouse event的属性

  • pageX： 返回页面左上角的相对距离，包括滚动距离(IE9+)
  • offsetX： 这个属性很容易混淆，所以加粗了。网上有的博客说这个属性与pageX表现一样，其实是错误的，正确的说是相对于父容器的偏移，与下文的offsetTop/Height类似(IE6+)
  • screenX： 返回距离屏幕左上角的距离
  • clientX： 距离浏览器可视区域左上角的距离(IE6+)

为了直观地看出区别，小生画了一张图：

HTMLelement的属性

  • offsetTop： 相对offsetParent元素的top偏移
  • offsetLeft： 相对offsetParent元素的left偏移(IE Edge支持)
  • offsetParent： 相对其定位的父元素 (IE Edge支持)
  • offsetHeight： 渲染时的高度(IE Edge支持)
  • offsetWidth： 渲染时的宽度(IE Edge支持)

js中浮点数的精度问题

发表于 2017-05-25

计算机中浮点数是用二进制存储的，很多浮点数的二进制表示是无限位的（例：0.1的二进制表示为0.00011001100110011001100110011001100110011001100110011…），所以浮点数计算过程中必定需要涉及边界（精度）选择的问题。在强类型语言中，这类问题通常会在语言内部处理掉。

在JavaScript中所有数字（包括整数）都是以64位浮点数形式储存。作为一门弱类型语言，JavaScript浮点数运算时没有实现精度自动处理的机制，因而会产生以下的问题：

0.1 + 0.2 === 0.3 //false

解决方案1：toFixed(digit)来设定比较的位数。

(0.1 + 0.2).toFixed(2) === 0.3.toFixed(2) //true 要注意的是toFixed返回的是String类型

但如果是未知长度的浮点数之间的比较呢？一拍脑门想到的的方法是先toString()再从字符串中解析出最大小数位n，再设定toFixed(n)来比较。这样做似乎把简单问题复杂化了，下面有一种更好的解决方案。

解决方案2：全统一转化成字符串，去掉小数点，再转换成number类型进行比较

function f2num(f) {
    return Number(f.toString().replace('.', ''));
}

f2num(0.1) + f2num(0.2) === f2num(0.3) //true

惰性函数定义

发表于 2017-05-24

惰性函数定义也称“自定义函数（self-definition function）”，当函数只需要执行一次初始化时（有点类似于单例模式）可以使用该模式。能减少程序的重复工作，提高性能。

但该模式有一些要注意的地方：

  1. 原始函数上定义的属性在”惰性定义“过程之后会全部丢失
  2. 如果自定义函数赋值给其他变量，那么该变量将永远不会执行“惰性定义”部分。
//举个例子
let foo = () => {
  console.log('la');
  foo = () => {
    console.log('lala');
  }
}
foo.la = "property"

let bar = foo;

然后我们首先调用bar()再调用foo()


bar(); //la
bar(); //la
console.log(bar.la); //property

foo(); //lala
console.log(foo.la); //undefined

然后我们首先调用foo()再调用bar()

console.log(bar === foo); //true
foo();  //la
console.log(foo.la); //undefined
foo();  //lala

console.log(bar === foo); //false
bar();  //la
bar();  //la
console.log(bar.la); //property

分析上面的例子我们可以知道，bar()内的函数指针指向foo()的原始函数，所以会一直执行原始函数部分，在执行原始函数的过程中又会不断地重写foo()的函数指针。反过来，foo()自身“惰性定义”并不会影响bar()的函数指针，所以导致了一些看似迷惑的现象。
